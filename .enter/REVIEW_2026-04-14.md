## 二、开发阶段回顾

### 阶段 1：AI 聊天核心（提交 1-8）
- 搭建 React 19 + TypeScript + Vite + Tailwind 项目骨架
- 实现 Perplexity 风格的搜索首页 + 对话界面
- 接入 Claude Sonnet 4.5，通过 Edge Function 实现 SSE 流式响应
- 实现思考过程可视化（thinking block 折叠展示）

**经验**：SSE 流式解析需要处理多种 event 类型（message_start / content_block_delta / tool_use 等），必须严格按照 API 文档逐一处理。

### 阶段 2：国际化（提交 9-11）
- 集成 react-i18next + 浏览器语言自动检测
- 完成中英文翻译覆盖

**踩坑**：
1. i18next 默认开启 Suspense，导致白屏 → 需设置 `useSuspense: false`
2. 中文 JSON 中不能使用中文引号 `""` → 改用 `「」`

### 阶段 3：文档集合管理（提交 12-15）
- 创建数据库表（document_collections + documents）+ RLS 策略
- 实现本地文件上传（TXT/MD/HTML/JSON/CSV/XML/YAML）
- 实现 URL 内容抓取（fetch-url-content Edge Function）
- 文档内容作为 system prompt 注入 AI 对话

**经验**：fetch-url-content 必须添加超时保护，否则慢速网站会导致整条请求链挂起。

### 阶段 4：修炼系统（提交 16-18）
- 设计 10 个境界（凡人 → 大道至简），每个境界有独立颜色和悟道点阈值
- 实现每日打卡（心境 / 无为指数 / 道场 / 感悟 4 个维度）
- AI 根据打卡数据生成个性化修炼指导
- 使用 localStorage 持久化，支持连续打卡加成

**经验**：localStorage 方案简单高效，但新增字段时必须做向后兼容（`?? false`）。

### 阶段 5：响应式优化（提交 19-22）
- 修复移动端顶部按钮被浏览器地址栏遮挡（top-4 → top-16）
- 统一按钮尺寸和图标大小
- 优化建议提示卡片的移动端布局

**教训**：移动端真机测试不可省略，模拟器无法反映浏览器 UI 元素的实际遮挡。

### 阶段 6：联网搜索（提交 23-25）
- 创建 web-search Edge Function（DuckDuckGo HTML 解析，免费无 API Key）
- 在 ai-chat 中实现 Tool Calling：AI 自动判断是否需要搜索
- 获取搜索结果后抓取前 2 个 URL 全文，作为 tool_result 回传 AI
- 前端展示来源卡片（标题 + 域名 + 摘要）

**踩坑**：
1. 流超时（stream timeout: no data received for 2 minutes）
   - 根因：fetch-url-content 内层 `fetch()` 无超时，慢速网站导致整条流挂起
   - 修复：三层超时防护（搜索 15s + 外层 URL 10s + 内层 fetch 8s）+ keepalive 进度消息
2. URL 获取数量从 3 减至 2，在速度和信息量之间取得平衡

### 阶段 7：导航栏重构（提交 26-27）
- 创建左侧固定导航栏（桌面端 w-64 固定，移动端抽屉式）
- 统一所有功能入口（新对话 / 文档集合 / 修行打卡 / 修炼指南）

### 阶段 8：修炼教程（提交 28-29）
- 5 步互动式新手引导（欢迎 → 境界 → 心境 → 实践 → 奖励）
- 完成教程奖励 50 悟道点
- 导航栏添加「修炼指南」入口

### 阶段 9：UI 精细化（提交 30-34）
- 联网搜索开关从侧边栏独立按钮 → 集成到搜索框底部工具栏
- Globe 图标 + ON 标签，视觉直觉更强
- 修复 NavigationSidebar 残留 props 导致的 "Failed to fetch" 错误
- 清理冗余代码和未使用的 import

**教训**：重构 props 接口时，必须同步更新所有调用点，否则会导致运行时崩溃。

## 五、关键技术决策

### 1. DuckDuckGo HTML 解析 vs 付费搜索 API
- **选择**：DuckDuckGo HTML 端点 + 正则解析
- **原因**：零成本、无需 API Key、足够满足需求
- **取舍**：稳定性略低于官方 API，但可通过超时 + 降级处理

### 2. localStorage vs 数据库（修炼数据）
- **选择**：localStorage
- **原因**：修炼系统是单机体验，无需多设备同步；避免增加数据库复杂度
- **风险**：清除浏览器数据会丢失进度（可后续升级为数据库存储）

### 3. 联网搜索开关位置
- **v1**：侧边栏独立按钮 + 首页浮动按钮
- **v2（当前）**：集成到搜索框底部工具栏
- **原因**：搜索开关与搜索输入强相关，放在一起更直觉、减少界面元素

### 4. Tool Calling 串行 vs 并行 URL 获取
- **选择**：串行获取（每个 URL 前后发送 keepalive）
- **原因**：并行虽快但无法发送进度消息，容易触发流超时
- **效果**：正常 17s、最坏 43s，远低于 120s 超时阈值

## 六、遇到的问题与解决方案

| # | 问题 | 根因 | 解决方案 |
|---|------|------|----------|
| 1 | i18n 白屏 | React Suspense 在 i18n 未就绪时渲染空白 | `useSuspense: false` |
| 2 | JSON 解析失败 | 中文全角引号 `""` 干扰 JSON | 改用 `「」` |
| 3 | 移动端按钮被遮挡 | 浏览器地址栏占用顶部空间 | `top-4` → `top-16` |
| 4 | 流超时 2 分钟 | fetch-url-content 内层 fetch 无超时 | 三层超时 + keepalive |
| 5 | Failed to fetch | 重构后残留旧 props 传入已更新组件 | 同步删除调用点的旧 props |
| 6 | GitHub 推送 403 | Token 权限不足 / 非组织成员 | 创建 classic token + 推送至个人仓库 |

## 八、总结

道研助手从零到发布历经 9 个开发阶段、34 次提交，核心经验：

1. **渐进式构建**：从 AI 聊天核心出发，逐步叠加功能，每阶段可验证
2. **超时是流式 API 的头号敌人**：所有外部 fetch 必须设超时 + keepalive
3. **重构时全链路检查**：删除 props/接口后，必须搜索所有调用点同步更新
4. **移动端不能只靠模拟器**：真机浏览器 UI 元素会遮挡固定定位元素
5. **简单方案优先**：localStorage 胜过过度设计的数据库方案；DuckDuckGo HTML 胜过付费 API
